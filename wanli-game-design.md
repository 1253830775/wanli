# AI 万历穿越文字冒险 — 游戏设计文档

## 1. 游戏概述

一款由 AI 驱动的历史穿越文字冒险游戏。玩家穿越成明神宗朱翊钧，从隆庆六年（1572 年）十岁登基开始，通过**上朝**这一核心场景，凭借现代知识做出决策，改变历史走向。游戏时间跨度覆盖完整万历朝 47 年（1572-1619），以萨尔浒之战为终点。

### 核心体验

- **场景聚焦**：所有玩法集中在朝堂之上——奏报、问对、裁决
- **改变历史**：预设历史事件是素材而非剧本，玩家的每个决策都可能改写历史走向
- **信息博弈**：皇帝不是全知的，需要找对人、问对问题才能获取关键信息
- **长线叙事**：47 年时间线，NPC 会老、会死、会换人，朝堂格局持续演变

## 2. 核心玩法循环

### 2.1 单次朝会流程

每次朝会包含 **2~4 个议题**，每个议题是一个完整的小闭环：

```
奏报阶段
  某大臣上奏（AI 生成事件描述 + 背景）
  ↓
问对阶段
  玩家可 @大臣 追问细节（1~3 轮自由问答）
  不同大臣提供的信息可能不同（信息差机制）
  ↓
裁决阶段
  玩家做出决策（自由文本输入 或 AI 提供 2~3 个选项）
  ↓
反应阶段
  大臣们的即时反应（赞同/反对/沉默）
  NPC 态度根据裁决结果动态变化
  ↓
→ 下一个议题
```

朝会结束后：

```
退朝
  ↓
AI 生成本次朝会小结
  包含"史书对照"：据《明史》记载，此事本应……然而在你的干预下……
  ↓
时间推进（快进数月~数年）
  ↓
下次朝会开始
  AI 摘要交代上次朝会至今各决策的后果
  ↓
进入新一轮奏报
```

### 2.2 议题来源（混合模式）

每次朝会的议题由两部分组成：

| 类型 | 数量 | 来源 | 说明 |
|------|------|------|------|
| 预设历史事件 | 1 个 | 历史事件库 | 考成法推行、高拱罢免、张居正夺情、国本之争、三大征、萨尔浒... |
| AI 随机政务 | 1~2 个 | AI 根据世界状态生成 | 黄河水患、边关告急、科举舞弊、地方叛乱... |

预设事件的触发条件由世界状态和 NPC 态度决定。如果玩家提前改变了相关条件，该事件可能不再触发，AI 生成替代事件。

### 2.3 朝堂冲突（克制型）

每个预设议题自带**阵营倾向**：

```
议题：是否推行考成法
  张居正 → 力推（支持方）
  高拱   → 反对（反对方）
  冯保   → 观望（根据皇帝态度站队）
```

大臣之间的冲突通过**措辞和态度值变化**体现，保持君臣礼数，暗流涌动而非当庭争吵。玩家裁决后同时影响多个 NPC 的态度——支持一方，另一方不满。每次决策都有代价。

## 3. 指标体系（三层设计）

### 3.1 第一层：面板常驻（皇帝"直觉感知"）

始终显示在 UI 状态面板上，玩家随时可见。

| 指标 | 说明 | 初始值 |
|------|------|--------|
| 国库（两） | 国家财政，影响军事、赈灾等能力 | 2,000,000 |
| 民望 | 百姓对朝廷的满意度 | 65 |
| 帝威 | 皇帝个人权威，影响政令推行和朝堂秩序 | 50（十岁登基，权威有限） |

帝威的特殊意义：
- 帝威高 → 朝堂秩序井然，大臣恭敬
- 帝威低 → 群臣放肆，政令难行
- 帝威归零 → 触发逼宫事件

### 3.2 第二层：需询问大臣获取（信息差机制）

不在面板显示，玩家需要通过 @大臣 主动询问才能获取。

| 指标 | 信息来源 | 说明 |
|------|----------|------|
| 边防态势 | 兵部尚书 / 边将 | 边境安全状况、外敌动向 |
| 官僚效率 / 腐败度 | 首辅 / 都察院 | 考成法效果、吏治状况 |
| 派系力量对比 | 冯保（内廷）/ 首辅（外朝） | 朝堂各派势力消长 |

不同大臣提供的信息可能**不完全一致**——冯保告诉你的和张居正告诉你的可能有出入。这是朝堂博弈的一部分。

### 3.3 第三层：隐藏指标（AI 内部追踪）

不直接展示数字，通过 NPC 言行和叙事体现。

| 指标 | 说明 |
|------|------|
| NPC 态度值 | 每个 NPC 对皇帝的态度（0~100），通过措辞和行为体现 |
| NPC 关系矩阵 | N×N 矩阵，追踪 NPC 之间的盟友/对立关系 |
| 因果事件链 | 蝴蝶效应的完整记录，玩家通过叙事感受而非列表查看 |
| 历史偏离度 | 当前时间线与真实历史的差异程度，结局评价时使用 |

## 4. 时间线与节奏

### 4.1 变速制

47 年时间线按历史节奏分为密集期和平稳期：

| 时期 | 年份 | 朝会次数 | 特征 |
|------|------|----------|------|
| 密集期 | 1572-1582 | 8~10 次 | 隆庆托孤 + 张居正改革 |
| 平稳期 | 1582-1592 | 4~6 次 | 万历亲政 + 国本之争 |
| 密集期 | 1592-1600 | 6~8 次 | 万历三大征 |
| 平稳期 | 1600-1615 | 3~5 次 | 矿税之祸 + 怠政 |
| 密集期 | 1615-1619 | 5~8 次 | 梃击案 + 萨尔浒 |

总计约 **25~35 次朝会**。

- 密集期：每次朝会间隔数月，AI 摘要覆盖较短时间
- 平稳期：每次朝会间隔 1~3 年，AI 摘要覆盖较长时间

### 4.2 罢朝机制

玩家可以主动选择**罢朝**（不上朝），这是一种高风险策略：

| 要素 | 说明 |
|------|------|
| 触发方式 | 玩家主动选择 |
| 代理机制 | 当前首辅代为裁决所有议题 |
| 次数限制 | 无限制 |
| 收益 | 某些指标可能反而好转（如首辅能力强，政务处理得当） |
| 风险 | 帝威下降、首辅权力坐大、信息获取减少 |

罢朝期间的信息反馈：
- 下次朝会时，AI 摘要交代罢朝期间发生的事
- 首辅只汇报部分事务，玩家可追问细节
- 被委以重任的 NPC 忠诚度上升，被冷落的 NPC 不满

### 4.3 首辅更替

首辅更替本身作为一个**朝会事件**处理：

```
触发条件：首辅死亡 / 被罢免 / 辞官 / 帝威过低被弹劾
  ↓
朝会议题：谁来接任首辅？
  ↓
候选人由 AI 根据当前 NPC 列表生成（各有立场和能力）
  ↓
玩家裁决（或罢朝时由前任首辅指定继任者）
```

罢朝期间首辅权力极大。如果首辅在罢朝期间死亡，继任者可能是玩家不喜欢的人——朝堂已被前任首辅经营成了自己的地盘。

## 5. NPC 系统

### 5.1 NPC 生命周期（完全动态）

NPC 的登场和退场完全由世界状态和 NPC 态度决定，玩家可以改变每个人的命运。

| 机制 | 说明 |
|------|------|
| 预设退场 | 关键 NPC 有历史死亡/退场年份作为参考，但玩家可改变条件阻止退场 |
| 预设登场 | 新 NPC 在特定年份/条件下登场（申时行、戚继光、李如松、魏忠贤...） |
| AI 生成 NPC | 非关键位置（六部侍郎、御史等）由 AI 根据历史背景生成 |
| 退场事件 | NPC 退场作为朝会事件或摘要内容呈现 |

### 5.2 NPC 关系矩阵

维护一个 N×N 的态度矩阵，追踪所有 NPC 之间的盟友/对立关系：

```
         张居正   高拱   冯保   李氏   ...
张居正     -      -20    +30    +10
高拱      -20      -     -40    +5
冯保      +30     -40     -     +15
李氏      +10     +5     +15     -
...
```

- 每次裁决后更新多个值（支持一方，其盟友对皇帝态度上升，其对手对皇帝态度下降）
- NPC 之间的关系也随事件动态变化
- AI 生成朝堂对白时参考此矩阵，自然形成派系博弈

### 5.3 NPC 人格档案

每个关键 NPC 维护结构化档案：

```json
{
  "id": "zhang_juzheng",
  "name": "张居正",
  "traits": ["严厉", "忠诚", "务实", "雄才大略"],
  "knowledgeBoundary": ["不知后世之事", "不知玩家穿越身份"],
  "attitude": {
    "toEmperor": 75,
    "history": [{"event": "玩家支持考成法", "delta": +10}]
  },
  "relations": {
    "gao_gong": -20,
    "feng_bao": 30,

  },
  "status": "顾命大臣 / 首辅",
  "faction": "改革派",
  "alive": true,
  "historicalDeathYear": 1582
}
```

## 6. 记忆系统（三层递进压缩）

### 6.1 架构总览

```
┌─────────────────────────────────────────────────┐
│  实时层：当前朝会上下文                            │
│  ─ 当前议题、已发生的问对、大臣反应                 │
│  ─ 生命周期：单次朝会，朝会结束即清除                │
│  ─ 注入方式：直接放入 user prompt                  │
├─────────────────────────────────────────────────┤
│  近记忆层：最近 3~5 次朝会摘要                      │
│  ─ 每次朝会结束后由 AI 生成 200 字摘要               │
│  ─ 包含：关键决策、NPC 态度变化、世界状态变化         │
│  ─ 注入方式：放入 system prompt                    │
├─────────────────────────────────────────────────┤
│  远记忆层：超长期摘要 + 结构化数据                   │
│  ─ 每 5 次朝会，AI 将近记忆再压缩成一段总摘要         │
│  ─ 加上：NPC 关系矩阵、历史偏离度清单、因果链         │
│  ─ 注入方式：放入 system prompt                    │
└─────────────────────────────────────────────────┘
```

### 6.2 压缩流程

```
朝会结束
  ↓
AI 生成朝会摘要（200 字以内）
  包含：议题、决策、NPC 态度变化、世界状态变化
  ↓
存入近记忆（保留最近 3~5 条）
  ↓
每 5 次朝会
  ↓
AI 将近记忆压缩为一段总摘要
  ↓
存入远记忆（同时更新 NPC 关系矩阵、历史偏离度清单）
  ↓
近记忆清空，重新开始积累
```

### 6.3 Prompt 注入策略

每次朝会生成叙事时，system prompt 中包含：

```
远记忆（1 段总摘要）
  + 近记忆（3~5 段朝会摘要）
  + 当前世界状态
  + NPC 档案（在场 NPC）
  + NPC 关系矩阵（在场 NPC 之间的子矩阵）
  + 当前事件节点
  + 知识图谱子图
```

Token 消耗始终可控：system prompt 中永远只有 **1 段远记忆 + 3~5 段近记忆**，不会随游戏时长无限膨胀。

### 6.4 决策日志

每次裁决记录结构化数据：

```json
{
  "round": 5,
  "courtSession": 3,
  "topic": "是否推行考成法",
  "decision": "支持张居正，全面推行",
  "affectedNpcs": [
    {"npcId": "zhang_juzheng", "attitudeDelta": +10},
    {"npcId": "gao_gong", "attitudeDelta": -15}
  ],
  "worldStateDelta": {
    "treasury": -50000,
    "publicSupport": +3,
    "imperialAuthority": +5
  },
  "historicalOutcome": "历史上考成法于万历元年推行",
  "deviation": "与历史一致"
}
```

## 7. 世界状态更新机制

### 7.1 AI 驱动的状态变更

LLM 在每轮叙事结束时输出一个结构化 JSON 块：

```
【状态变更】
{
  "treasury": -50000,
  "publicSupport": +3,
  "imperialAuthority": +5,
  "npcAttitudes": {
    "zhang_juzheng": +10,
    "gao_gong": -15
  },
  "npcRelations": {
    "zhang_juzheng-gao_gong": -5
  }
}
```

后端解析此 JSON 并应用到世界状态和 NPC 系统。

### 7.2 状态变更的叙事反馈

世界状态变化不直接显示数字，而是通过下次朝会的叙事体现：

- 国库减少 → AI 摘要中提到"工部奏报银两吃紧"
- 民望上升 → 朝会上有大臣奏报"各地百姓称颂圣德"
- 帝威下降 → 大臣奏报时措辞变得不那么恭敬

## 9. 结局系统

### 9.1 正常结局（1619 年萨尔浒之后）

AI 根据全部远记忆生成一段**史书评价**，包含：

- **庙号**：根据治国表现（如"神宗"是真实庙号，改变历史后可能不同）
- **谥号**：根据一生功过
- **史官点评**：200~500 字的文言文评价，涵盖内政、军事、用人、历史影响

示例：

> 《明史·神宗本纪》赞曰：帝幼冲即位，委政居正，考成一条，国用充裕。
> 及亲政之后，……（根据玩家实际表现生成）

### 9.2 逼宫事件（软性 Game Over）

当指标恶化到临界点时触发：

| 触发条件 | 场景 |
|----------|------|
| 帝威归零 | 大臣联名上书要求退位 |
| 国库破产 + 民变 | 亡国危机 |
| 军队忠诚度归零 | 兵变 |

逼宫场景作为特殊朝会处理：
- 玩家必须在朝会上力挽狂澜（拉拢关键大臣、做出让步、展示决心）
- 成功 → 续命，但帝威上限降低
- 失败 → 提前结束，生成"亡国之君"评价

### 9.3 结局类型

| 结局 | 条件 | 评价基调 |
|------|------|----------|
| 中兴之主 | 各项指标优秀 + 历史大幅正向偏离 | 千古一帝 |
| 守成之君 | 指标平稳 + 历史偏离不大 | 中规中矩 |
| 亡国之君 | 逼宫失败 / 指标全面崩溃 | 昏君 |
| 权臣傀儡 | 帝威极低但未触发逼宫 | 被权臣操控的木偶皇帝 |
| 偏安之主 | 军事失败但内政尚可 | 文治有余而武功不足 |
| 改革先驱 | 大幅改变历史 + 制度创新 | 超越时代的改革者 |

## 10. 交互设计

### 10.1 玩家输入方式

| 操作 | 输入方式 | 示例 |
|------|----------|------|
| 自由发言 | 直接输入文本 | "朕以为此法甚善，当推行之" |
| 追问大臣 | @NPC名 + 问题 | "@张居正 考成法具体如何考核？" |
| 选择选项 | 点击 AI 提供的选项 | 裁决阶段 AI 给出 2~3 个选项 |
| 罢朝 | 点击"罢朝"按钮 | 跳过本次朝会 |
| 退朝 | 输入"退朝"或点击按钮 | 结束本次朝会 |

## 11. 技术实现要点

### 11.1 需要新增/改造的后端模块

| 模块 | 说明 |
|------|------|
| 朝会编排器 | 管理单次朝会的议题流程（奏报→问对→裁决→反应） |
| 事件触发引擎 | 根据世界状态和 NPC 态度判断预设事件是否触发 |
| 记忆压缩服务 | 朝会结束后调用 AI 生成摘要，管理三层记忆 |
| NPC 关系矩阵 | N×N 矩阵的存储和更新 |
| 状态变更解析器 | 解析 AI 输出的结构化 JSON 状态变更 |
| 时间推进器 | 管理时间线推进、NPC 生命周期事件 |
| 首辅代理逻辑 | 罢朝时由首辅 NPC 代为裁决 |
| 逼宫触发器 | 监控指标临界点，触发逼宫事件 |
| 结局生成器 | 游戏结束时调用 AI 生成史书评价 |

### 11.2 需要改造的现有模块

| 模块 | 改造内容 |
|------|----------|
| ContextAssembler | 重写 prompt 构建逻辑，注入三层记忆 + NPC 关系矩阵 |
| NarrativeService | 接入状态变更解析、记忆压缩、事件触发 |
| NpcService | 接入态度更新、关系矩阵更新 |
| WorldStateService | 接入状态变更，支持增量更新 |
| EventNodeService | 从简单关键词匹配改为条件触发引擎 |
| LLMClient | 增加结构化输出解析能力 |

### 11.3 需要新增的前端组件

| 组件 | 说明 |
|------|------|
| 议题进度条 | 显示当前议题的阶段（奏报→问对→裁决） |

| 朝会摘要弹窗 | 展示上次朝会至今的 AI 摘要 |
| 史书对照提示 | 在叙事中高亮"史书对照"内容 |

## 12. 附录：设计决策记录

| # | 问题 | 决策 | 理由 |
|---|------|------|------|
| 1 | 指标体系 | 三层：面板常驻 + 问大臣获取 + 隐藏 | 模拟帝王信息困境，强化 @NPC 交互价值 |
| 2 | 游戏场景 | 局限在上朝 | 天然的信息汇聚 + 决策闭环 |
| 3 | 游戏节奏 | 连续剧制 | 长线叙事张力 |
| 4 | 朝会之间 | AI 摘要 | 简洁，保持上朝节奏 |
| 5 | 单次朝会结构 | 奏报→问对→裁决→反应 | 每个议题一个小闭环 |
| 6 | 议题来源 | 混合：1 预设 + 1~2 随机 | 兼顾历史纵深和世界活力 |
| 7 | 历史可改性 | 完全可改 | 预设事件是素材不是剧本 |
| 8 | 朝堂冲突 | 克制型 | 暗流涌动比吵架更有味道 |
| 9 | 蝴蝶效应感知 | 摘要中加入史书对照 | 让玩家感知到自己在偏离历史 |
| 10 | 终点 | 1619 萨尔浒 | 完整 47 年万历朝 |
| 11 | NPC 生命周期 | 完全动态 | 玩家可改变每个人的命运 |
| 12 | 朝会节奏 | 变速制 | 符合历史节奏，高潮迭起 |
| 13 | 罢朝 | 策略选择，首辅代理，无限制 | 不上朝本身是有意义的决策 |
| 14 | 首辅更替 | 作为朝会事件处理 | 首辅是谁是游戏中最重要的变量 |
| 15 | 结局评价 | AI 生成史书评价 | 比分数更有余韵 |
| 16 | Game Over | 软性恶化，逼宫场景 | 不中断体验，制造戏剧高潮 |
| 17 | 记忆系统 | 三层递进压缩 | Token 可控，信息不丢 |
| 18 | NPC 关系 | N×N 关系矩阵 | 驱动派系博弈 |
